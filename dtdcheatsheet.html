Introduction to DTDs

A Document Type Definition (DTD) defines the legal building blocks of an XML document. This section introduces the two main types of DTDs and how to declare them. Understanding this is the first step to creating well-structured and validatable XML.

Internal DTD

Declared inside the XML file using the `<!DOCTYPE>` declaration. It's best for small, self-contained documents.

<!DOCTYPE root_element [
  <!-- DTD declarations go here -->
]>

External DTD

Defined in a separate `.dtd` file. This is preferred for reusability across multiple XML documents.

<!DOCTYPE root_element SYSTEM "filename.dtd">

Declaring Elements

Elements are the core building blocks of XML. This section explains how to define elements and their content models using the `<!ELEMENT>` declaration. You can interact with the tabs below to explore different types of content models.

Empty Elements

Elements with no content.

<!ELEMENT br EMPTY>

Parsed Character Data (#PCDATA)

Elements that contain text.

<!ELEMENT name (#PCDATA)>

Any Content

Elements that can contain any combination of other elements.

<!ELEMENT note ANY>

Defining Structure and Cardinality

You can precisely control the order and number of child elements.

Sequence ( , ): `<!ELEMENT note (to, from, heading, body)>` - Children must appear in this order.
Choice ( | ): `<!ELEMENT contact (email | phone)>` - Allows one of several elements.

Occurrence Indicators

+

One or more

<!ELEMENT p (b+)>

*

Zero or more

<!ELEMENT p (b*)>

?

Zero or one

<!ELEMENT p (b?)>

Mixed Content

Elements that can contain both text (#PCDATA) and other elements. The `#PCDATA` keyword must come first, and children must be optional and grouped.

<!ELEMENT p (#PCDATA|b|i|u)*>

This allows an element like `

This is bold and italic text.

`

Declaring Attributes

Attributes provide extra information about elements. The `<!ATTLIST>` declaration defines attributes for an element, including their types and default values. Explore the different types and defaults below.

Attribute Types

CDATA: Simple string of text.
ID / IDREF / IDREFS: For creating unique identifiers and references.
NMTOKEN / NMTOKENS: A valid XML name token.
Enumerated: A list of possible values, e.g., `(check|cash|credit)`.

Default Values

#REQUIRED: The attribute must be included.
#IMPLIED: The attribute is optional.
#FIXED "value": The attribute value is fixed.
"default_value": Provides a default if not specified.

Example ATTLIST Declaration

<!ATTLIST book   isbn      CDATA #REQUIRED   status    (available|checked_out) "available"   in-series IDREF #IMPLIED>

Declaring Entities

Entities are like variables or shortcuts for standard text, special characters, or even other DTD declarations. They help reduce repetition and make DTDs more maintainable.

Internal Entity

A simple shortcut for text defined within the DTD.

<!ENTITY copyright "© 2025">

Usage in XML: `&copyright;`

External Entity

Includes content from an external file.

<!ENTITY footer SYSTEM "footer.html">

Usage in XML: `&footer;`

Parameter Entity

A variable used only within the DTD itself.

<!ENTITY % text "#PCDATA">
<!ELEMENT p (%text;)>

Reference in DTD: `%text;`

Complete Example

This section provides a complete, side-by-side example of an external DTD and an XML file that uses it. This demonstrates how all the concepts—elements, attributes, and entities—work together to define and validate a document's structure.

notes.dtd

<!-- DTD for a simple notes collection -->
<!ELEMENT notes (note+)>

<!ELEMENT note (to, from, heading, body)>
<!ATTLIST note
    id ID #REQUIRED
    priority (high|low) "low">

<!ELEMENT to (#PCDATA)>
<!ELEMENT from (#PCDATA)>
<!ELEMENT heading (#PCDATA)>
<!ELEMENT body (#PCDATA)>

<!ENTITY company "My Awesome Company">

document.xml

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE notes SYSTEM "notes.dtd">

<notes>
  <note id="n01" priority="high">
    <to>Tove</to>
    <from>Jani</from>
    <heading>Reminder</heading>
    <body>Don't forget me this weekend!</body>
  </note>
  <note id="n02">
    <to>Jani</to>
    <from>Tove</from>
    <heading>Re: Reminder</heading>
    <body>I will not</body>
  </note>
</notes>

Validation

Validation is the process where an XML parser checks a document against its DTD to ensure it conforms to the defined rules. This final step is crucial for ensuring data integrity and consistency before the data is processed by an application.

Why Validate?

A validating parser will report an error if the XML document violates any DTD rules, such as:

  • A required attribute is missing.
  • An element appears in the wrong order or is not allowed.
  • An element contains invalid child elements.
  • An attribute has a value not listed in an enumeration.

This process guarantees that the XML data is structured correctly, preventing errors in data exchange and application logic.
